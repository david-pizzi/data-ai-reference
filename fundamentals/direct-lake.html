Direct Lake

High-performance semantic model storage mode that reads directly from OneLake Delta tables
Module 4 Reference
Sections: What Is Direct Lake Storage Modes Comparison How It Works — Transcoding Framing Automatic Updates DirectQuery Fallback Capacity Guardrails When to Use Direct Lake Knowledge Check
What Is Direct Lake
Power BI semantic model table storage mode — a distinct mode alongside Import and DirectQuery that determines how data is accessed and stored
Optimized for large volumes from Delta tables — reads Parquet files directly from OneLake without importing or copying data into the semantic model
Fast loading into memory for interactive analysis — transcodes columnar data from Parquet into the in-memory VertiPaq engine on demand
Ideal for large lakehouses and warehouses — designed for enterprise-scale analytics where data resides in OneLake Delta tables

Key benefits:

Import-like performance — queries run against in-memory columnar data, delivering interactive speeds comparable to Import mode
No full data refresh required — eliminates lengthy ETL-style import cycles; data is read directly from source files
Automatic sync with OneLake — reflects changes to underlying Delta tables without manual intervention (when automatic updates are enabled)

How It Works — Transcoding

Direct Lake loads data into the VertiPaq in-memory engine by transcoding Parquet column segments on demand:

Columns loaded on-demand from OneLake — when a DAX query references a column, the engine fetches the corresponding Parquet column segments and transcodes them into VertiPaq format
Only needed columns loaded — not the entire dataset; if a query touches 3 of 50 columns, only those 3 are loaded, minimizing memory and I/O
Columns stay resident in memory — once transcoded, column data remains cached in memory for subsequent queries, delivering Import-like performance
Eviction reasons — columns are evicted from memory when a refresh (framing) occurs, when they have not been used recently, or under memory pressure from the capacity
Parquet files organized by columns — the columnar layout of Parquet aligns perfectly with VertiPaq’s columnar storage, making transcoding efficient without row-by-row transformation
Framing

Framing is the process that determines which version of the Delta table data the semantic model reads:

Point-in-time control — a frame defines the exact set of Delta log files and Parquet data files the model should reference, providing a consistent snapshot
Low-cost metadata operation — framing completes in seconds, not minutes, because it only updates metadata pointers rather than moving or copying data
Analyzes Delta log to drop only affected segments — the engine inspects the Delta transaction log to identify which in-memory column segments are stale, evicting only those
Incremental framing — dictionaries (encoding maps) are usually not dropped during a frame update; only the affected data segments are refreshed, preserving most cached data
Trigger options — framing can be initiated manually in the Fabric portal, automatically when OneLake data changes, on a scheduled cadence, or programmatically via the TMSL/XMLA refresh API
Automatic Updates
Enabled by default — new Direct Lake semantic models automatically detect changes to the underlying Delta tables in OneLake
Auto-reflects OneLake changes — when data in the lakehouse or warehouse is updated, the model automatically frames the new version without manual intervention
Can be disabled for manual framing control — in scenarios requiring strict version control or coordinated refresh windows, automatic updates can be turned off so framing is triggered only on demand
Automatic page refresh for reports — Power BI reports connected to Direct Lake models can leverage automatic page refresh to reflect the latest data to end users without requiring a manual browser refresh
DirectQuery Fallback

When Direct Lake cannot serve a query from its in-memory cache, it may fall back to DirectQuery mode:

Only applies to Direct Lake on SQL endpoints — fallback routes queries through the SQL analytics endpoint of the lakehouse or warehouse; this mechanism is exclusive to SQL-based Direct Lake
Fallback triggers — occurs when SQL-level security (RLS/CLS) is enforced, when the model references SQL views, or when capacity guardrails (max rows, max file count) are exceeded
Falls back to SQL analytics endpoint queries — the engine redirects the query to the SQL endpoint, which processes it against the data warehouse or lakehouse tables directly
Slower performance than Direct Lake — DirectQuery fallback queries are processed by the SQL engine rather than the in-memory VertiPaq engine, resulting in higher latency
DirectLakeBehavior property — controls whether fallback is allowed; set to Automatic (default, allows fallback), or DirectLakeOnly (disables fallback, returns errors instead)
Direct Lake on OneLake does NOT support fallback — models using Direct Lake on OneLake (without a SQL endpoint) have no fallback path; queries that exceed guardrails will fail rather than degrade

When to Use Direct Lake
IT-driven analytics with lake-centric architectures — best suited for organizations centralizing data in OneLake lakehouses or warehouses managed by data engineering teams
Large volumes in OneLake — when datasets are too large for traditional Import mode refresh windows, or when data freshness requirements exceed scheduled refresh capabilities
Fast loading, frequent refresh, efficient capacity use — Direct Lake minimizes capacity consumption by loading only queried columns and avoiding full data copies
Import still relevant for self-service analysts — users connecting to non-OneLake sources, requiring calculated tables, or working with smaller datasets may still prefer Import mode
OneLake integration for Import models — Import models can also benefit from OneLake by using it as a staging area, but they still require scheduled refresh to bring data into memory
Consider prototype/POC first — before migrating large Import models to Direct Lake, validate performance with a proof of concept to ensure guardrails and data structure align with requirements
